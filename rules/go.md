# Go 代码审查专项规则

## 1. 并发
- [ ] `goroutine` 是否有对应的 `WaitGroup` / `context` 控制？
- [ ] `channel` 是否正确关闭（避免 panic）？
- [ ] `select` 是否有 `default` 避免阻塞？
- [ ] `sync.Pool` 是否在频繁分配场景使用？
- [ ] `mutex` 是否最小化临界区？

## 2. 错误处理
- [ ] 错误是否包装而非直接返回（`fmt.Errorf("...: %w", err)`）？
- [ ] 是否检查所有 error 返回值？
- [ ] `panic` 是否仅在不可恢复场景使用？
- [ ] `defer` 顺序是否正确（LIFO）？

## 3. 性能
- [ ] 字符串拼接是否用 `strings.Builder`？
- [ ] 切片扩容是否预分配（`make([]T, 0, n)`）？
- [ ] `interface{}` 是否必要（影响编译优化）？
- [ ] 反射是否在高频路径使用？

## 4. 内存
- [ ] 闭包是否捕获循环变量（Go < 1.22 陷阱）？
- [ ] 大对象是否逃逸到堆（避免不必要的指针）？
- [ ] `nil` 切片 vs 空切片 `[]T{}` 是否区分场景？

## 5. 项目规范
- [ ] 是否通过 `go vet` / `golint` / `staticcheck`？
- [ ] 包名是否简洁（无下划线、无驼峰）？
- [ ] 导出的标识符是否有注释？
- [ ] `context.Context` 是否作为第一个参数传递？
