# 数据库/SQL 审查专项规则

## 1. SQL 质量
- [ ] 是否用 `SELECT *`（应指定字段）？
- [ ] `JOIN` 是否缺少 `ON` 条件导致笛卡尔积？
- [ ] `WHERE` 条件是否对索引字段做函数操作（导致索引失效）？
- [ ] `LIKE '%xxx%'` 是否导致全表扫描？
- [ ] `IN` 子句元素是否过多（>1000 应分批或换 `JOIN`）？
- [ ] `ORDER BY` 是否配合 `LIMIT`（避免大数据排序）？
- [ ] 是否用 `UNION ALL` 替代 `UNION`（去重有性能开销）？
- [ ] 批量插入是否用 `INSERT ... VALUES (...), (...)`？

## 2. 索引
- [ ] 新查询是否走索引（`EXPLAIN` 检查）？
- [ ] 联合索引是否遵循最左前缀？
- [ ] 索引字段是否做类型转换（隐式转换导致失效）？
- [ ] 是否重复索引（如已有 (a,b) 又建 (a)）？
- [ ] 大表加索引是否用 `ALGORITHM=INPLACE`？

## 3. 事务
- [ ] 事务范围是否最小化？
- [ ] 是否死锁风险（不同事务加锁顺序不一致）？
- [ ] 是否长事务（应及时提交）？
- [ ] 读操作是否用 `SELECT FOR UPDATE`（不必要的锁）？

## 4. 数据一致性
- [ ] 金额是否用 `DECIMAL` 而非 `FLOAT/DOUBLE`？
- [ ] 时间是否用 `DATETIME`/`TIMESTAMP` 而非字符串？
- [ ] 枚举是否用 `TINYINT` 或 `VARCHAR` 而非 `ENUM` 类型？
- [ ] 软删除是否统一用 `deleted_at` 字段？
- [ ] 外键是否加索引？

## 5. ORM 使用
- [ ] 是否 N+1 查询（关联查询未 `fetch join`/`preload`）？
- [ ] 批量更新是否用 `updateMany` 而非循环单条？
- [ ] 乐观锁是否用版本号/时间戳？
